Implementation: Photo Report

New blueprint over the existing headless generator. No PyQt5.

What it does

Builds a branded photo-report PDF from a folder of job images plus claim information. Images are resized and JPEG-compressed to fit a max file size; the PDF carries company branding, logo, header, footer, and one image per page with auto-detected EXIF dates.

Existing code to call (headless, no Qt)

  • restoration_common.PhotoReportPDF(output_path, job_info, company_config, logo_path=None) then .generate(image_files, progress_callback=None, max_size_mb=...).
  • restoration_common.load_companies(), get_company_by_id(), find_logo(), get_image_files(), generate_output_filename().
  • restoration_common.fetch_job_info_from_url(url) for CRM prefill.

job_info dict keys

customer_name, claim_number, job_location, job_id, photo_title, photo_date (optional; auto-detected from EXIF if blank).

Target

tools/photo_report/
  __init__.py     blueprint, url_prefix=/photo-report
  routes.py       GET form; POST -> PhotoReportPDF.generate -> send_file(pdf)
  forms.py        field set + validation
  templates/      form (layout seed: mockup photo-report page)

Web form -> generator mapping

  • Company dropdown -> company_config via get_company_by_id.
  • Job fields -> the job_info dict above.
  • Photos: <input type=file multiple> uploaded to a temp dir -> the image_files list. (The desktop folder picker becomes a multi-file upload.)
  • Max file size (MB) -> max_size_mb.
  • "Fetch from CRM" -> fetch_job_info_from_url, prefills the job fields. Network feature: see graceful degradation.

Output

Stream the PDF back with send_file(..., as_attachment=True); filename from generate_output_filename. No server-side retention needed.
