Maintainability and LLM-navigability

The structure is designed so a future edit lands in a small, findable file.

STRUCTURE.md is the map

Read it first. One row per meaningful path: its responsibility and "edit here when you want to change X." Examples: tools/iws/static/js/calculator.js for the coverage math; toolbox/registry.py to add or remove a tool; tools/documents/templates for invoice fields.

Single sources of truth

  • toolbox/registry.py for the set of tools (tiles, nav, registration).
  • core/templates/base.html + core/static/css/toolbox.css for chrome and styling, defined once; tools extend rather than duplicate.
  • toolbox/config.py for paths and flags.

Conventions

  • Predictable names so the edit target is found by path: HTTP in routes.py, logic in *_ops.py, forms in forms.py, PDF layout reused from restoration_common.
  • Around 300 lines per file; split by concern. Any unavoidably long file carries # === SECTION: name === banners that grep finds.
  • Every generated artifact (this spec file, any bundled asset) starts with a header naming its source and rebuild command, and is never hand-edited.

Why not one big file

The earlier self-contained mockup was 586 KB. That form is fine as a read-only artifact but a poor place to edit. The real app keeps normal multi-file templates; the single-file form survives only as this generated spec.
